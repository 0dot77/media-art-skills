# media-art-skills

이 프로젝트는 미디어아트 제작 업체(스튜디오)를 위한 Claude Code 스킬팩이다.

## 스킬 명령어

### 리서치/소싱
- `/model-search` — 3D 에셋 검색 (Sketchfab, CGTrader 등)
- `/reference-collect` — 프로젝트 레퍼런스 수집
- `/vendor-search` — 외주 업체/견적 조사
- `/hardware-search` — 장비 스펙 비교 검색

### 프로젝트 관리
- `/estimate` — 견적서 생성
- `/timeline` — 제작 일정표/마일스톤
- `/tech-spec` — 기술 사양서
- `/proposal` — 기획서/제안서

### 제작 파이프라인
- `/render-check` — 렌더 파이프라인 검증
- `/content-pipeline` — 미디어 에셋 관리 (해상도, 포맷, 코덱)
- `/display-calc` — 디스플레이 설치 계산
- `/color-check` — 색공간/캘리브레이션 검증

### 현장/설치
- `/site-survey` — 현장 답사 체크리스트/보고서
- `/install-guide` — 설치 매뉴얼
- `/troubleshoot` — 트러블슈팅 가이드
- `/maintenance` — 유지보수 일정/체크리스트

## 문서 저장 규칙

### 저장 경로
스킬 실행 시 **처음 한 번** 저장할 프로젝트 폴더를 질문한다:
> "문서를 어디에 저장할까요? (폴더 경로를 지정해주세요)"

지정된 경로 아래에 문서를 저장한다. 경로가 없으면 자동 생성한다. 이후 같은 세션에서는 동일 경로를 재사용한다. 이전에 저장한 문서가 해당 경로에 있으면 자동으로 읽어 컨텍스트로 사용한다.

### 출력 포맷
모든 문서는 **`.docx` (Word)** 형식으로 저장한다. 기획자가 바로 클라이언트에게 전달할 수 있는 실무 포맷이다.

- `.docx` 변환에는 `python-docx` 라이브러리를 사용한다
- 변환 스크립트: `scripts/md-to-docx.py`
- 마크다운 원본도 함께 저장하여 버전 관리가 가능하도록 한다
- 저장 구조:
  ```
  {지정된 경로}/
  ├── TECH-SPEC.docx       ← 클라이언트 전달용
  ├── TECH-SPEC.md         ← 원본 (버전 관리용)
  ├── ESTIMATE.docx
  ├── ESTIMATE.md
  └── ...
  ```

## 데이터 핸드오프 규격

스킬 간 데이터는 프로젝트 폴더 내 `.md` 파일을 통해 전달된다. 각 스킬은 아래 테이블에 따라 **특정 섹션/필드**를 읽고 쓴다.

### 핸드오프 매트릭스

| 생산 스킬 (Writer) | 출력 문서 | 핵심 출력 필드 | 소비 스킬 (Reader) | 읽는 필드 |
|---|---|---|---|---|
| `/site-survey` | SITE-SURVEY.md | 공간 치수, 가용 전력(A/W), 네트워크, 반입 경로, 천장 하중 | `/display-calc` | 벽면 크기, 투사 거리, 천장 높이 |
| | | | `/tech-spec` | 가용 전력, 네트워크 포트, 환경 조건 |
| | | | `/install-guide` | 접근 제약, 반입 경로, 작업 가능 시간 |
| `/display-calc` | DISPLAY-CALC.md | 출력 해상도, 프로젝터 모델/수량, 밝기, throw distance, 블렌딩 설정 | `/tech-spec` | 디스플레이 구성 전체 |
| | | | `/content-pipeline` | 콘텐츠 출력 해상도, FPS |
| | | | `/estimate` | 장비 모델, 수량 |
| `/tech-spec` | TECH-SPEC.md | 장비 목록(모델/수량/소비전력), 네트워크 구성, 콘텐츠 사양, 총 소비전력 | `/estimate` | 장비 목록 → 장비비 산출, 소비전력 → 전력 비용 |
| | | | `/timeline` | 장비 리드타임, 제작 복잡도 |

| | | | `/maintenance` | 장비 목록 → 점검 항목, 소모품 |
| | | | `/troubleshoot` | 시스템 구성 → 예상 문제 목록 |
| `/estimate` | ESTIMATE.md | 카테고리별 비용, 총액, 지급 스케줄, 외주 항목 | `/proposal` | 예산 요약, 지급 스케줄 |
| | | | `/timeline` | 외주 항목 → 리드타임 반영 |
| `/timeline` | TIMELINE.md | 단계별 기간, 마일스톤, 데드라인 | `/estimate` | 기간 → 인건비 산출 |
| | | | `/proposal` | 일정 요약 |
| `/content-pipeline` | CONTENT-SPEC.md | 코덱, 해상도, FPS, 색공간, 스토리지 요구량 | `/render-check` | 검증 기준 (해상도, 코덱, FPS, 색공간) |
| | | | `/color-check` | 콘텐츠 색공간 |

### 핸드오프 규칙

1. **읽기**: Context Check 단계에서 위 테이블의 "읽는 필드"에 해당하는 값을 파싱한다. 값이 존재하면 사용자에게 보여주고 확인만 받는다 (중복 질문 금지).
2. **쓰기**: 출력 문서에 "핵심 출력 필드"를 일관된 마크다운 헤딩/테이블 구조로 작성하여 다음 스킬이 파싱할 수 있게 한다.
3. **누락 시**: 필요한 입력 문서가 없으면 해당 스킬을 먼저 실행하도록 안내한다. 사용자가 원하면 수동 입력도 허용한다.
4. **충돌 시**: 두 문서의 값이 충돌하면 (예: DISPLAY-CALC의 밝기 ≠ TECH-SPEC의 밝기) 사용자에게 어느 값을 사용할지 질문한다.

## 검증 레이어

모든 스킬은 계산/추천 결과를 `references/` 데이터와 **자동 대조**한다. 생성만 하고 끝나는 것이 아니라, 결과의 실현 가능성을 검증한다.

### 검증 규칙

| 검증 대상 | 참조 데이터 | 검증 내용 | 결과 표시 |
|-----------|-----------|----------|----------|
| 프로젝터 throw ratio | `hardware-db.md` 프로젝터 테이블 | 계산된 throw ratio가 선택 모델의 실제 범위 내인지 | ✅ 범위 내 / ❌ 범위 초과 + 대안 제시 |
| 프로젝터 밝기 | `hardware-db.md` | 필요 밝기 ≤ 모델 스펙 밝기 | ✅ 충분 / ⚠️ 부족 (N lm 모자람) |
| LED 시청 거리 | `hardware-db.md` LED 테이블 | 실제 관람 거리 ≥ 피셀 피치 기반 최소 시청 거리 | ✅ 적정 / ⚠️ 너무 가까움 |
| 미디어서버 출력 | `hardware-db.md` 미디어서버 테이블 | 필요 출력 수/해상도 ≤ 서버 최대 출력 | ✅ 가능 / ❌ 초과 |
| 센서 감지 범위 | `hardware-db.md` 센서 테이블 | 설치 공간 크기 ≤ 센서 유효 범위 | ✅ 커버 / ⚠️ 범위 부족 |
| 코덱 호환성 | `codec-reference.md` | 선택 코덱이 미디어서버에서 지원되는지 | ✅ 호환 / ❌ 비호환 + 대안 |
| 색공간 일관성 | `color-spaces.md` | 파이프라인 전체 색공간/감마 일치 여부 | ✅ 일치 / ❌ 불일치 + 조치 |
| 전력 가용성 | SITE-SURVEY.md | 총 소비전력 ≤ 현장 가용 전력 | ✅ 충분 / ❌ 부족 (N W 초과) |
| 반입 가능성 | SITE-SURVEY.md | 장비 크기 ≤ 출입문/엘리베이터 크기 | ✅ 통과 / ❌ 불가 |

### 검증 출력 형식

각 스킬은 출력 문서 마지막에 `## Validation Report` 섹션을 포함한다:

```
## Validation Report

| 항목 | 기준값 | 계산값 | 결과 |
|------|--------|--------|------|
| Throw Ratio (Panasonic PT-RQ25K) | 1.2–2.0:1 | 1.7:1 | ✅ 범위 내 |
| 필요 밝기 vs 모델 밝기 | 25,000 lm | 18,000 lm | ✅ 충분 |
| 가용 전력 vs 소비전력 | 30A (6,600W) | 5,200W | ✅ 여유 1,400W |
| 센서 범위 vs 공간 폭 | 6m (RealSense D455) | 8m | ⚠️ 범위 부족 — LiDAR 권장 |

⚠️ 주의사항 1건: 센서 범위가 공간 폭보다 작습니다. Livox Mid-360 (40m)으로 변경을 권장합니다.
```

### 검증 실패 시 행동

- ⚠️ **경고**: 결과를 출력하되 경고와 대안을 명시한다. 사용자가 진행 여부를 결정한다.
- ❌ **오류**: 결과를 출력하되 해당 항목을 강조 표시하고, 수정하지 않으면 현장 문제가 발생할 수 있음을 경고한다.

## 프로젝트 타입 감지

PROJECT.md의 내용을 보고 제작 유형을 자동 감지한다:

- "프로젝션 매핑" / "projection mapping" → 프로젝션 매핑
- "LED" → LED 디스플레이
- "인터랙티브" / "interactive" → 인터랙티브 설치
- "키네틱" / "kinetic" → 키네틱 아트 / 피지컬 컴퓨팅
- "몰입형" / "immersive" → 이머시브 공간
- 감지 불가 시 사용자에게 질문

### 프로젝트 타입별 스킬 분기

각 스킬은 감지된 프로젝트 타입에 따라 **다른 절차와 중점 항목**을 적용한다:

| 스킬 | 프로젝션 매핑 | LED 디스플레이 | 인터랙티브 | 키네틱 | 이머시브 |
|------|-------------|-------------|-----------|--------|---------|
| `/display-calc` | throw ratio, 블렌딩, 곡면 워핑 | 피셀 피치, 캐비닛, 시청 거리 | 해상도, 센서-디스플레이 매핑 | 디스플레이 크기/무게 | 다면 프로젝션, 시야각 |
| `/site-survey` | 투사 거리, 주변광, 벽면 재질/색상 | 구조물 하중, 전력 용량 | 센서 설치 위치, 관객 동선 | 천장 리깅, 하중 한계, 안전 거리 | 바닥/벽/천장 전면, 음향 반사 |
| `/tech-spec` | 프로젝터+미디어서버 중심 | LED 컨트롤러+프로세서 중심 | 센서+미디어서버+네트워크 중심 | 모터 컨트롤+안전 시스템 | 멀티채널 출력+공간 오디오 |
| `/estimate` | 프로젝터, 렌즈, 마운트 | LED 패널, 컨트롤러, 구조물 | 센서, 개발비, 테스트 | 모터, 가공, 안전 장치 | 대면적 장비, 음향 |
| `/install-guide` | 정렬/블렌딩 캘리브레이션 | 캐비닛 조립, LED 프로세서 설정 | 센서 캘리브레이션, 인터랙션 테스트 | 기구부 조립, 안전 테스트 | 다면 정렬, 몰입도 테스트 |
| `/troubleshoot` | 블렌딩 어긋남, 밝기 불균일 | 데드 픽셀, 모아레 | 센서 오인식, 레이턴시 | 모터 과열, 소음, 안전 정지 | 이음새 불일치, 음향 반사 |
| `/maintenance` | 프로젝터 필터/렌즈 청소 | LED 패널 검수, 밝기 균일도 | 센서 청소, SW 업데이트 | 기계부 윤활, 벨트/기어 점검 | 전면 캘리브레이션 재확인 |

각 SKILL.md 내에서 프로젝트 타입에 따라 분기되는 세부 절차가 정의되어 있다.

## 피드백 루프

프로젝트 완료 후 현장 결과를 기록하여 `references/` 데이터의 정확도를 높인다.

### 프로세스

```
현장 설치/운영
    ↓
/maintenance 실행 시 FIELD-LOG 기록 안내
    ↓
{프로젝트 폴더}/FIELD-LOG.md 에 기록
    ↓
축적된 피드백을 references/ 에 반영
```

### FIELD-LOG.md 구조

```markdown
# Field Log — {프로젝트명}

## 장비 피드백
| 장비 | 스킬 제안 | 실제 결과 | 차이 원인 | references 반영 |
|------|----------|----------|----------|---------------|
| Panasonic PT-RQ25K | 18,000 lm 충분 | 주변광으로 부족 | 창문 차광 미흡 | hardware-db.md에 "반옥외 환경 시 1.5배 권장" 추가 |
| RealSense D455 | 6m 범위 | 4m까지만 안정 | 흑색 의상 인식 불가 | hardware-db.md 센서 비고에 추가 |

## 현장 조건 피드백
| 항목 | 사전 조사 | 실제 | 개선점 |
|------|----------|------|--------|
| 가용 전력 | 30A | 실제 20A (다른 장비 공유) | site-survey에 "전용 회로 확인" 항목 추가 |

## 코덱/콘텐츠 피드백
| 항목 | 스펙 가이드 | 실제 결과 | 개선점 |
|------|-----------|----------|--------|
| HAP Q 4K 60fps | 재생 정상 | SSD 온도 과열 (45분 후) | codec-reference.md에 장시간 재생 주의사항 추가 |
```

### 피드백 반영 규칙

1. **기록 시점**: `/maintenance` 또는 `/troubleshoot`를 실행할 때 FIELD-LOG.md가 없으면 생성을 안내한다. 이미 있으면 새 항목을 추가한다.
2. **반영 시점**: FIELD-LOG.md에 3건 이상의 동일 패턴 피드백이 축적되면, `references/` 해당 파일에 반영을 제안한다.
3. **반영 방식**: references/ 파일의 기존 데이터는 수정하지 않고, 비고/주의사항 컬럼에 현장 경험 데이터를 추가한다. 사용자 승인 필수.
4. **프로젝트 간 학습**: 새 프로젝트에서 `/display-calc` 등을 실행할 때, 이전 프로젝트의 FIELD-LOG.md가 있으면 유사 조건의 피드백을 참고한다.

## 행동 원칙

1. **실무 관점** — 클라이언트 납품과 현장 설치를 전제로 한다.
2. **수치 기반** — 비용, 일정, 스펙은 구체적 수치로 제시한다 (px, lm, A, ₩).
3. **검증 필수** — 모든 계산/추천 결과를 `references/` 실제 데이터와 대조한다. 생성 후 반드시 Validation Report를 출력한다.
4. **비파괴적** — 기존 파일을 직접 수정하지 않는다. 새 파일 생성이 기본.
5. **한 번에 하나의 질문** — 사용자에게 여러 질문을 동시에 하지 않는다.
6. **이중 언어** — 클라이언트 문서는 한/영 모두 생성 가능. 사용자의 언어를 따른다.
7. **피드백 순환** — 현장 결과를 기록하고, 축적된 데이터로 references/를 개선한다.
